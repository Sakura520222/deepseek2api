# DeepSeek2API 项目概述

## 项目简介

DeepSeek2API 是一个纯 Node.js 构建的 DeepSeek Web 控制台与 OpenAI 兼容桥接服务。它将本地用户体系、DeepSeek 账号绑定、API Key 管理、原生代理调试和管理后台集成在同一个可直接运行的项目中，无需第三方运行时依赖即可部署。

## 技术栈

**后端：** 纯 Node.js 原生 HTTP 服务，无框架，数据持久化仅使用 `data/app.json` 单文件 JSON。  
**前端：** 原生 JavaScript + CSS + HTML，静态资源由服务端托管，不依赖前端框架或打包工具。  
**鉴权与安全：** DeepSeek 账号 token 管理、自动刷新；PoW（工作量证明）挑战自动求解，调用 WebAssembly；API Key 生成与校验均在服务端原生实现。  
**部署环境：** Node.js 18+，需能访问 `chat.deepseek.com`、`cdn.deepseek.com` 及 `fe-static.deepseek.com`。

## 架构设计与关键决策

- **前后端同仓分离**：`public/` 为前端，`src/` 为后端，按功能领域拆分文件，通过 `app.js` 协调前端模块，路由层仅做分发。
- **分层架构**：`routes` 处理请求分发与参数提取；`services` 集中业务逻辑；`storage` 封装 JSON 文件读写，保障一致性；`utils` 提供通用辅助。
- **依赖方向明确**：路由 → 桥接服务 → `completion-core` → 请求模块。桥接层采用统一的 `normalize → resolve → collect/stream` 模式，但 system/tool prompt 构建方式尚未完全统一，prompt 组装与错误格式的共享仍需继续推进。
- **工具调用多策略兜底**：`tool-prompt.js` 具备多种解析策略应对多样化输出，但调用侧的门控限制导致兜底策略无法生效，形成"策略存在但无法触发"的隐蔽问题。
- **共享核心强化趋势**：`completion-core.js` 正成为工具调用解析逻辑的汇聚点，常量、标记、检测函数均集中于此，有利于复用但需警惕膨胀。
- **SSE 解码与业务过滤未解耦**：`deepseek-sse.js` 中解码逻辑与信号过滤耦合较紧，长期看应将 SSE 解码、帧类型识别、业务过滤分层处理。
- **模型自动生成机制**：前端 `CHAT_MODELS` 声明基础模型，后端 `BASE_OPENAI_MODELS` 声明映射，`createModelVariant` 自动衍生变体（如 `-search`）。审查时必须理解生成逻辑边界，避免遗漏变体的属性继承问题。
- **认证与轮询透明化**：DeepSeek token 失效自动刷新、PoW 自动求解、API Key 多账号轮询均在 `services` 层闭环，上层无感。
- **配置外部化**：仅通过 `.env` 管理可变配置，`.env.example` 提供模板。
- **流式/非流式路径双轨**：`responses-bridge` 等桥接服务存在流式与非流式两套输出构建路径，尾部组装逻辑极易产生结构性不一致，已识别为重复犯错高发区。

## 已知问题与注意事项

- **工具调用门控逻辑**：`openai-bridge.js` 中若 `tools` 为空会跳过 `extractToolCalls`，导致不发送 `tools` 字段的客户端无法触发工具调用。修复时需移除提前返回的条件，同时保证返回的 `tool_calls` 包含 `id` 等必要字段。类似的条件启用逻辑可能散布在其他调用点，遇到兼容性问题时需全面排查。
- **隐式控制流脆弱性**：`extractFragmentText` 等函数通过字符串类型优先处理实现隐式分支，虽已添加注释，但控制流仍脆弱。未来应推荐显式 `if/else if` 分支或提前返回结构，而非仅靠注释。
- **公共函数提取需提升健壮性标准**：提取到公共模块的函数（如 `checkForToolCallMarker`）其实现（如松散字符串匹配）在私有上下文中可用，但作为公共 API 需更强的契约，审查应关注其健壮性。
- **SSE 过滤依赖底层字段风险**：过去用 `!payload.v?.response?.fragments` 过滤已完成帧，属依赖深层嵌套字段。现已改为 `typeof payload.v === "string"` 的类型特征判断，更稳定。后续审查遇到依赖深层嵌套字段的过滤，应优先引导改用更稳定的类型/特征判断。
- **外部客户端适配风险**：上游客户端行为差异可能触发连锁适配问题。分析兼容性 Issue 时必须追溯触发条件与上游文档，将逆向抓包确认为前置验证步骤。适配难度分层：OpenClaw 简单 < Codex 中等 < Claude Code 复杂。
- **多桥接一致性问题**：多个桥接服务在 system/tool prompt 构建、认证解析、错误转换上缺乏统一策略，易引入偏差。流式响应的尾部输出构建是重复犯错高发区。
- **前端渲染泄漏风险**：DeepSeek 原始 `<think/>` 标签可能在 Web 控制台 SSE 流或第三方客户端中直接暴露给用户，需排查 `public/deepseek-message.js` 的渲染逻辑。
- **模型支持更新依赖外部格式**：新增模型不仅需更新前端模型列表和后端模型映射，还可能涉及 `model_type` 值和请求头版本号，部分字段需逆向抓包确认。
- **外部 API 演进风险**：拟支持的 OpenAI Responses API 格式仍在演进，Anthropic Messages API 的工具调用格式与现有 `tool-prompt.js` 存在显著差异，引入新接口需预留自适应适配层。
- **数据存储单点**：全量状态集中在 `data/app.json`，需注意并发写入安全与备份恢复。
- **API 参数冲突**：当参数语义冲突（如 `tool_choice=required` 但无 `tools`）时，应返回 4xx 错误而非静默忽略。
- **防御性门控移除风险**：移除 `if (requestOptions.tools)` 等门控后，工具调用解析逻辑将对所有请求无条件生效，可能引入误触发脆弱性。每次移除均需附风险评估与测试策略。
- **API 契约完整性**：任何允许无 `tools` 定义下提取工具调用的修改，必须附带对 API 契约的明确审查。

## 审查中发现的重要模式

- **工具调用 Issue 根因定位**：不能只看解析逻辑，必须溯源到触发条件。门控式跳过是典型陷阱，未来类似问题应优先排查此类条件分支。
- **常量逐步规范化与统一前缀**：从硬编码字符串到局部常量，再到风格统一的前缀命名（如 `TOOL_CALL_*`），可显著降低误用风险。审查应推动以统一前缀建立语义簇。
- **条件精确化优于逻辑依赖**：用类型判断（如 `typeof`）替代依赖深层嵌套字段的过滤，是降低脆弱性的有效方向。
- **冗余防御代码识别**：审查任何 `if` 条件组合，特别是带可选链的，需反复推演其在不同调用上下文中的真假值，防止“永远为真”或“永远为假”的逻辑冗余。此类代码无害但会误导维护者。
- **SSE 信号判别原则**：禁止仅凭文本内容（如 `"FINISHED"`）过滤数据帧，必须依据结构特征（如 `payload` 中特定字段缺失或类型差异）区分，并加注释说明信号帧特征。
- **流式解析器分支顺序依赖注释**：对于如 `extractFragmentText` 存在强分支顺序依赖的函数，要求关键分支处添加注释，声明当前变量类型与状态。
- **索引安全把关**：调用 `parseJsonFrom` 等起始位置函数时，须在传入前校验 `indexOf` 结果不为 -1。
- **外部客户端适配模式**：客户端行为差异会触发防御性代码绕过功能，内部分析时应关注“基于标准 SDK 行为的隐藏假设”，并验证请求抓包。
- **模型同步类 Issue 特征**：总是涉及前端模型列表、后端模型映射、请求头版本号与可能的 `model_type` 调整，可固化为检查清单。
- **自动生成机制审查模式**：通过 `createModelVariant` 等函数衍生的变体，必须显式确认生成规则不会覆盖关键属性，建议附上全量模型列表逐一核对。
- **多桥接一致性格查**：存在多个桥接服务时，必须检查 prompt 构建、认证解析、错误转换的统一策略。
- **流式输出事件完整性**：对于流式处理后构建的最终响应，必须验证 `output` 数组结构与流式事件中的 `output_index`、`content_part` 一一对应。
- **分层思考法**：将问题拆解为“数据格式（API）”和“表现格式（UI）”两个正交层面，先定义边界再分析。必须追问“这个错误是用户看到的，还是代码打印的？”
- **模糊标题处理**：类似“新功能”“改进”等无信息量标题，必须提取正文语义后再分类与重复检测。
- **重复检测透明化**：必须给出相似度分数与摘要对比，禁止仅输出模糊结论。信息不足时需明确声明“需人工复核”。
- **外部不确定性纳入优先级**：外部 API 不稳定性应作为优先级权衡因子；未确认的新模型需产 Warning。
- **标签与仓库体系对齐**：推荐标签时需注意仓库实际使用的标签体系。适配类 Issue 必备 `compatibility`，前端展示类需补 `area/frontend`。
- **增量审查的信任但验证**：即使前一阶段问题标记为已修复，仍需逐一核对修复情况并审视是否引入新结构性风险。每轮必须逐条列出前轮问题及关闭证据。

## 团队约定与规范

- **模块化职责**：单文件承担单一职责，前端按功能命名，后端分层调用，路由不写入业务逻辑，存储层封装文件操作。
- **结构化 Issue 标题**：建议采用 `[分类][优先级] 动作 + 具体对象` 格式，例如 `[compatibility][medium-high] 添加 OpenClaw 工具调用适配支持`。标题应点明核心发现和行动项。
- **工具调用相关修改**：涉及工具调用解析或提取的变更，必须同步检查 `openai-bridge.js`、`tool-prompt.js` 及所有触发条件，确保门控逻辑与多策略兜底一致。
- **流式处理相关修改**：任何涉及流式响应的 PR，必须显式对比增量事件与最终对象的一致性，避免遗漏尾部事件或结构脱节。
- **新模型引入四维检查清单**：
  1. 前端模型列表字段完整性
  2. 后端 `BASE_OPENAI_MODELS` 映射一致性
  3. 请求头/版本号常量是否需要更新
  4. 全局搜索相关硬编码取值，确保无旧取值范围残留
- **新接口/模型适配**：需提供影响文件清单、格式差异对比、工作量估算与外部风险提示。外部 API 未确认时必须产出 Warning，附带待确认清单。
- **防御性门控移除规范**：移除条件门控需附风险评估和测试策略，视同核心逻辑变更，不可简单归为“清理死代码”。
- **常量命名一致性规则**：当同一文件或同一语义域存在多个常量时，应使用统一前缀（如 `TOOL_CALL_*`）形成命名空间，审查中若发现孤立命名需提示。
- **SSE 数据帧过滤条件编写规范**：过滤条件不仅要求正确，还应通过注释说明“信号帧的结构特征”，避免未来维护者误放宽条件。
- **代码质量分级**：未使用变量与死代码清理应标记为 minor 或 suggestion，不应标记为 major 或 blocking，除非导致逻辑错误。建议在 lint 规则中启用 `no-unused-vars` 并设为 error。
- **调试环境变量规范**：所有 `DEBUG_*` 环境变量应仅用于日志输出，不得影响核心分支逻辑；新增此类变量时需说明其目的与范围。禁止调试日志完整记录用户输入，必须对 prompt 内容实施截断或哈希化，并提供 `DEBUG_SANITIZE` 选项。
- **工具参数字段映射规范**：对于多 API 参数名兼容（`parameters` / `input_schema` / `inputSchema`），约定标准化中间表示（统一为 `inputSchema`），在入口处规范转换，避免链式 fallback 散落各处。
- **多桥接共享逻辑提取**：任何在 ≥2 个桥接文件中出现的相同功能函数，必须提取到 `completion-core.js` 或新建共享模块，评审时将其标记为必要条件。
- **无构建依赖**：前端直接使用原生 ES Module 或全局脚本，后端以 CommonJS/ESM 原生运行，提交代码不得引入构建工具或 TypeScript。
- **数据操作**：所有对 `data/app.json` 的读写必须通过 `storage` 层，调用方不得直接操作文件系统。
