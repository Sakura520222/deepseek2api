# DeepSeek2API 项目概述

## 项目简介

DeepSeek2API 是一个纯 Node.js 构建的 DeepSeek Web 控制台与 OpenAI 兼容桥接服务。它将本地用户体系、DeepSeek 账号绑定、API Key 管理、原生代理调试和管理后台集成在同一个可直接运行的项目中，无需第三方运行时依赖即可部署。

## 技术栈

**后端：** 纯 Node.js 原生 HTTP 服务，无框架，数据持久化仅使用 `data/app.json` 单文件 JSON。  
**前端：** 原生 JavaScript + CSS + HTML，静态资源由服务端托管，不依赖前端框架或打包工具。  
**鉴权与安全：** DeepSeek 账号 token 管理、自动刷新；PoW（工作量证明）挑战自动求解，调用 WebAssembly；API Key 生成与校验均在服务端原生实现。  
**部署环境：** Node.js 18+，需能访问 `chat.deepseek.com`、`cdn.deepseek.com` 及 `fe-static.deepseek.com`。

## 架构设计与关键决策

- **前后端同仓分离**：`public/` 为前端，`src/` 为后端，按功能领域拆分文件，通过 `app.js` 协调前端模块，路由层仅做分发。
- **分层架构**：`routes` 处理请求分发与参数提取；`services` 集中业务逻辑；`storage` 封装 JSON 文件读写，保障一致性；`utils` 提供通用辅助。
- **条件化启用的防御性编码**：项目存在多处“有参数才启用”的门控逻辑，典型如 `openai-bridge` 中仅当请求携带 `tools` 参数时才提取工具调用，否则提前返回。这是导致某些客户端（如 OpenClaw，其能力检测时不发送 `tools`）工具调用失效的根因。
- **工具调用多策略兜底**：`tool-prompt.js` 具备多种解析策略应对多样化输出，但调用侧的门控限制导致兜底策略无法生效，形成“策略存在但无法触发”的隐蔽问题。
- **认证与轮询透明化**：DeepSeek token 失效自动刷新、PoW 自动求解、API Key 多账号轮询均在 `services` 层闭环，上层无感。
- **配置外部化**：仅通过 `.env` 管理可变配置，`.env.example` 提供模板。

## 已知问题与注意事项

- **工具调用门控逻辑**：`openai-bridge.js` 中若 `tools` 为空会跳过 `extractToolCalls`，导致不发送 `tools` 字段的客户端（如 OpenClaw）无法触发工具调用。修复时需移除提前返回的条件，同时保证返回的 `tool_calls` 包含 `id` 等必要字段，避免客户端报错。类似的条件启用逻辑可能散布在其他调用点，遇到兼容性问题时需全面排查所有调用链。
- **外部客户端适配风险**：上游客户端行为差异（如能力检测、格式差异）可能触发连锁适配问题。分析兼容性 Issue 时必须追溯触发条件与上游文档或版本信息。
- **模型支持更新依赖外部格式**：新增模型不仅需更新前端模型列表和后端模型映射，还可能涉及 `model_type` 值变化和请求头版本号，部分字段可能需要逆向抓包确认。
- **外部 API 演进风险**：拟支持的 OpenAI Responses API 格式仍在演进，Anthropic Messages API 的工具调用格式（`tool_use`）与现有 `tool-prompt.js` 存在显著差异，引入新接口需预留自适应适配层并关注上游变更。
- **数据存储单点**：全量状态集中在 `data/app.json`，需注意并发写入安全与备份恢复，避免数据损坏。

## 审查中发现的重要模式

- **工具调用 Issue 根因定位**：不能只看解析逻辑，必须溯源到触发条件（如 `tools` 参数是否存在）。门控式跳过是典型陷阱，未来类似问题应优先排查此类条件分支。
- **外部客户端适配模式**：客户端行为差异（如能力检测时不发送某参数）会触发防御性代码绕过功能，分析时应关注 Issue 中是否附带客户端名称、版本或上游链接。
- **模型同步类 Issue 特征**：总是涉及前端模型列表、后端模型映射、请求头版本号与可能的 `model_type` 调整，可固化为检查清单，提升分析效率。
- **模糊标题处理**：类似“新功能”“改进”等无信息量标题，必须提取正文语义后再进行分类与重复检测，禁止直接依据标题判断。
- **重复检测透明化**：当信息不足（如标题过短、描述含糊）时，不应轻易输出“可能是 #X 的重复”，应要求人工复核或附带相似度片段；自动重复标记必须有语义相似度依据。
- **外部不确定性纳入优先级**：外部 API 的不稳定性（如格式变动、退役截止日期）应作为优先级权衡因子，可在规则中自动提升或降低优先级并附加提醒。
- **标签与仓库体系对齐**：推荐标签时需注意仓库实际使用的标签体系，若无访问权限应给出通用推荐并提示“请根据仓库实际标签调整”。

## 团队约定与规范

- **模块化职责**：单文件承担单一职责，前端按功能命名（如 `auth-ui.js`），后端分层调用，路由不写入业务逻辑，存储层封装文件操作。
- **结构化 Issue 标题**：建议采用 `[分类][优先级] 动作 + 具体对象` 格式，例如 `[enhancement][medium] 添加 OpenClaw 工具调用适配支持`，便于检索与看板筛选。
- **工具调用相关修改**：涉及工具调用解析或提取的变更，必须同步检查 `openai-bridge.js`、`tool-prompt.js` 及所有触发条件，确保门控逻辑与多策略兜底一致。
- **新接口/模型适配**：需提供影响文件清单、格式差异对比、工作量估算与外部风险提示，并关注测试覆盖与对现有路由的干扰。
- **无构建依赖**：前端直接使用原生 ES Module 或全局脚本，后端以 CommonJS/ESM 原生运行，提交代码不得引入构建工具或 TypeScript。
- **数据操作**：所有对 `data/app.json` 的读写必须通过 `storage` 层，调用方不得直接操作文件系统。
