## 常见代码模式与审查要点
- **工具调用门控陷阱**：工具调用解析须覆盖无 tools 声明场景；新增解析器需与排除条件集成正确，要求压力测试。工具标记维护必须数组驱动，禁止硬编码链式判断，统一用 TOOL_CALL_ 前缀常量。
- **模型同步影响模块**：新增模型须检查前端列表、后端映射、请求头版本、model_type 四维一致；未公开 API 产出 Warning。
- **桥接服务一致性**：多桥接 prompt 构建、认证、错误转换须统一；≥2 个桥接出现相同功能函数必须提取到 completion-core.js。流式与非流式输出路径须对比结构。
- **流式输出尾部完整性**：收尾逻辑须验证最终 output 结构与事件对应；提取共享收尾函数；强制检查是否遗漏 output 项。
- **自动生成变体审查**：createModelVariant 衍生变体须确认不错误继承 thinkingEnabled。
- **边缘保护**：参数语义冲突（如 tool_choice=required 但无 tools）返回 4xx。
- **未使用变量与死代码**：标记 minor/suggestion；启用 lint no-unused-vars 为 error；删除函数前须全局搜索调用点，或靠 ESLint no-undef 阻断构建。
- **冗余防御代码**：if 条件组合及可选链须推演真假值，防止永远为真/假。
- **索引安全**：调用 parseJsonFrom 等起始位置函数前必须校验 indexOf 结果不为 -1。
- **SSE 信号判别**：禁止仅凭文本内容（如 "FINISHED"）过滤数据帧，必须依据结构特征或显式信号字段，并加注释说明帧结构。

## Issue 分析与优先级模式
- 适配类 Issue 推荐 compatibility 标签；展示类附加 area/frontend。
- 重复检测必须给出相似度分数与摘要对比，信息不足声明需人工复核。
- 外部依赖退役日期可提升优先级；功能阻断上调至 high。
- 分层分析区分数据格式（API）与表现格式（UI）。

## 经验教训与规范建议
- 多桥接重构易引入参数传递偏差；提取公共收尾函数避免流式遗漏。
- 涉及工具调用解析修改需补充无 tools 时兜底测试。
- 隐式控制流的注释修复是双刃剑，优先用显式 if/else if 或提前返回。
- 常量提取时预判扩散风险，通用语义应考虑导出。
- 条件精确化优于依赖深层嵌套字段，推荐用 typeof 类型判断。
- 修复不等于无懈可击，可能引入过度防御，审查须关注逻辑精简性。
- 新增公共函数须提升健壮性标准，脆弱匹配在公共 API 中不可接受。
- 调试日志脱敏：不得完整记录用户输入或对话历史，对 prompt 截断或哈希化，提供 DEBUG_SANITIZE 环境选项。
- 环境变量（DEBUG_*）仅用于日志，不得影响核心业务分支。
- 多桥接架构需尽早定义基类或规范管道，降低维护成本。

- 外部客户端适配与前端渲染
- 调试开关 DEBUG_TOOL_CALL 周边逻辑，确保不影响业务分支
