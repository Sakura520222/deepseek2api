## 常见代码模式与审查要点
- **工具调用触发条件陷阱**：当反馈“工具调用未生效”时，务必检查门控逻辑（如 `openai-bridge.js` 中仅当 `tools` 参数非空才执行提取的提前返回）。该仓库多处存在“条件化启用”的防御性编码，修改一处后应检查所有调用点。
- **模型同步影响模块**：新增模型时需系统性扫描四个区域：前端模型列表、后端模型映射、请求头版本号更新、`model_type` 可能变化；如遇外部 API 版本不明确，需提示可能需逆向工程确认。
- **多接口适配的复杂性**：引入新接口（如 Responses API、Anthropic Messages API）时，需显式识别格式差异、工具调用格式映射、流式处理差异，并评估外部 API 演进带来的维护风险。

## Issue 分析与优先级模式
- **优先级调权因子**：外部依赖退役日期可合理提升优先级（如旧模型退役临近）；外部 API 不稳定则应在高低之间加入“外部不确定性”折扣。
- **模糊标题处理**：严禁直接依据标题分类，必须提取正文语义生成摘要后再分类；标题改写应包含类型、优先级、动作和精确模型/接口名。
- **重复检测**：对于信息不足（<20 字）或仅有模糊结论的 Issue，不自动标记重复，明确提示“需人工复核”；必须附带相似度分数与摘要对比，否则降低结论置信度。
- **标签推荐**：通用标签（`enhancement`、`good first issue`）可用，但建议补充 `compatibility`/`adapter` 或 `area/api-compatibility` 等，以突出适配与兼容性；若无法获取仓库标签体系，需注明“请按实际标签调整”。`good first issue` 的推荐应附注指引，避免挫伤新手。

## 经验教训与规范建议
- 外部客户端（如 OpenClaw）适配时，需追踪上游能力检测行为与消息格式，验证返回格式（如 `tool_call_id`、`finish_reason`）是否能被客户端正确消费。
- 所有修改涉及工具调用的解析或提取，需同步更新或新增针对“无 tools 参数时兜底解析”的测试用例。
- 可行性评估中必须明确不确定性：对需要外部 API 验证的部分（如 `model_type` 取值）应标记为“需抓包确认”；工作量估算应拆解模块并与现有组件行数对比。
- 优先沉淀“模型添加”和“多接口适配”类 Issue 的分析模板，固化关键检查点与风险项。
