## 常见代码模式与审查要点
- **工具调用门控陷阱**：多处仅当 `tools` 非空才触发工具提取，修改逻辑后须复查所有调用点的门控条件。
- **模型同步影响模块**：新增模型需检查前端列表、后端映射、请求头版本号、`model_type` 四维一致性；对未公开 API 必须产出 Warning，列明待确认项。
- **桥接服务一致性**：多桥接的 system/tool prompt 构建、认证解析、错误转换须采用统一策略，不一致应要求收敛。
- **流式输出尾部完整性**：任何流式响应 PR 必须验证最终 output 结构与流式事件（`output_index`、`content_part`）对应；提取共享收尾函数，避免重复与遗漏。
- **自动生成变体审查**：`createModelVariant` 衍生 `-search` 变体时，需确认不会错误继承或覆盖 `thinkingEnabled`；审查时附上全量属性表格。
- **边缘保护**：参数语义冲突（如 `tool_choice=required` 但无 tools）须返回 4xx，不得静默忽略。
- **未使用变量与死代码**：标记为 minor/suggestion，除非构成逻辑错误否则不标为 major。

## Issue 分析与优先级模式
- **标签推荐**：兼容性/适配类问题必须推荐 `compatibility` 或 `area/api-compatibility`；无法获取标签体系时注明“请按实际标签调整”。
- **重复检测**：必须给出相似度分数与摘要对比，禁止仅输出模糊结论；信息不足时提示“需人工复核”。
- **优先级因子**：外部依赖退役日期可提升优先级；外部 API 不确定性应合理打折。
- **适配类 Issue**：审查时追问“基于标准 SDK 行为的隐藏假设”；防御性门控移除需明确设计权衡与测试策略。

## 经验教训与规范建议
- 多桥接重构易引入细节偏差，应特别留意参数传递与默认行为。
- 流式处理尾部构建是重复犯错高发区，提取公共收尾函数可避免维护遗漏。
- 外部 API 兼容性无法在审查中彻底解决，但应明确风险边界并给出验证步骤（如抓包）。
- 任何涉及工具调用解析的修改，需同步补充“无 tools 时兜底解析”的测试用例。
