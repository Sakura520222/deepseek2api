## 6. 经验教训

- **复杂多接口适配类型的 Issue 分析模板值得固化**：此类需求常涉及估工作量、识别组件复用、格式差异点列表，本案分析结构可复用为模板。  
- **工具系统差异需重点标注**：当涉及 Anthropic 与 OpenAI 工具调用差异时，分析应自动检查项目内是否有 tool 相关适配模块（本例提到了 tool-prompt.js），这点在未来分析中可自动化。  
- **重复检测必须透明化**：单纯输出“可能是 #X 的重复”价值有限，必须提供理由或内容相似度，否则应标注“未找到可靠重复”或降低置信度。  
- **标签推荐应与仓库标签体系对齐**：若无法访问仓库实际标签，应给出通用推荐并提示“请根据仓库实际标签调整”，避免直接给出可能不适配的标签。  
- **外部 API 演进风险应纳入优先级权衡**：Responses API 的不稳定性可导致维护成本增加，未来可在优先级判断中加入“外部不确定性”因子，适当升/降级。  

这些点后续可改进 Issue 分析助手的推理过程。
