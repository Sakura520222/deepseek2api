### 2. 发现的模式

- **模型扩展的标准流程**：前端 `CHAT_MODELS` 声明呈现属性，后端 `BASE_OPENAI_MODELS` 声明基础模型，再由 `createModelVariant` 自动生成 `-search` 变体。这种“声明一次，自动衍生”的模式降低了维护成本，但也意味着审查时必须理解生成逻辑的边界。  
- **一致性约束**：前端 `modelType` 必须与后端 `modelType` 对齐，且 `thinkingEnabled` 需在两端一致；`searchEnabled` 在前端显式声明，在后端由生成函数隐式设置。本次审查构建的对齐表格（包含生成变体）是验证该约束的高效方法。  
- **惯性假设**：代码假设 V4 与 V3 的 `model_type` 取值、请求路径、请求头完全一致。这是历史项目的常见模式，但在外部 API 未公开的情况下是高风险假设。

### 3. 规范完善建议

基于本次审查，可提炼以下审查规则（可写入项目记忆或审查提示）：

1. **新模型引入四维检查清单**：  
   - 前端模型列表字段完整性（`id, modelType, thinkingEnabled, searchEnabled` 等）  
   - 后端 `BASE_OPENAI_MODELS` 映射一致性  
   - 请求头/版本号常量的可能更新（如 `API_VERSION`、`model_type` 有效值枚举）  
   - 如有自定义映射（如 `openai-bridge` 中的 `modelType` 传递），需全局搜索相关引用，确保无硬编码旧取值范围。

2. **`createModelVariant` 的审查提示**：  
   当新增非标准变体（如 reasoner）时，需显式确认生成规则不会覆盖或错误继承 `thinkingEnabled` 等关键属性。建议在审查时附上生成后的全量模型列表（含所有属性）以便核对。

3. **外部 API 未确认时的标记规范**：  
   凡涉及新增外部 API 模型但缺乏官方文档验证时，必须产出 Warning，明确列出待确认项（如 `model_type`、`api_version`、`endpoint`），并建议抓包或等待文档。这已在本次审查中体现，可固化为模板。

4. **代码可读性风格**：  
   可添加审查规则：“当新增一组逻辑相关的模型超过 3 个时，建议在模型数组内用空行或注释分组，以提升可维护性”。

### 4. 经验教训

- **对自动生成机制的警惕**：表面上看只新增了 4 个基础模型，但实际通过 `createModelVariant` 扩展为 8 个前端条目。审查必须追踪这种隐式生成，否则容易漏掉 `-search` 变体的属性继承问题。本次审查通过长表格对比完全覆盖，这种实践值得保留。  
- **不确定性管理**：外部 API 兼容性无法在代码审查中彻底解决，但审查能够明确边界——我们确认了“如果 API 要求不同 `model_type` 则请求会被拒绝”这一后果，并给出了合理的验证建议。这种“识别风险、提出验证步骤”的审查风格，比强行要求改代码更务实。  
- **项目记忆的应用**：项目记忆中“新增模型时需系统性扫描四个区域”的条目在本审查中得到部分体现（检查了前端、后端映射，提示了 `model_type` 可能变化及请求头版本号）。后续可以强化成强制检查项，让审查者逐项打勾，避免仅凭经验疏忽。  
- **审查数据价值**：这次审查生成了一个对齐表格，这不仅能帮助作者验证，还可作为团队共识文档。可在审查模板中增加“前后端字段对齐矩阵”为可选项，方便复杂模型扩展时使用。

总的来说，这是一次高质量的审查，边界清晰、风险评估准确。反思后主要提升点在于：将“自动生成变体的审查”和“外部 API 不确定的标记”固化为标准化检查项，并在未来的审查中主动搜索可能存在的硬编码取值，进一步提升覆盖度。
