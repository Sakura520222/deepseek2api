### 6. 经验教训

#### 对类似 Issue 分析的可复用模式

1. **"新增 X API 桥接"类 Issue 的快速评估清单**：
   - 检查请求格式差异（normalize 层复杂度）
   - 检查认证方式差异（路由层改动）
   - 检查工具调用格式差异（是否触及 `tool-prompt.js`）
   - 检查流式格式差异（是否触及 `deepseek-sse.js`）
   - 检查是否加剧已知架构债（如"多桥接一致性"）
   这个清单本身可沉淀为分析模板。

2. **`good first issue` 标签的严格把关**：凡是涉及"多模块联动 + 格式转换 + 新认证方式"三个及以上维度的问题，即使代码量不大，也不应轻易推荐为 good first issue。应检查"认知负载"而非仅看行数。

3. **重复检测必须遵守契约**：项目记忆明确要求"相似度分数 + 摘要对比"，遗漏此信息会降低分析可信度。未来分析中应作为强制输出项。

#### 仓库特有问题模式

1. **`deepseek2api` 的桥接架构正在进入"数量膨胀期"**（3→4），项目记忆中"多桥接需尽早定义基类"的建议已经到了执行窗口。后续任何新增桥接的 Issue 都应附带"架构先升级再扩展"的技术建议，而不是继续复制粘贴桥接文件。

2. **流式解析是高度耦合的脆弱点**：`deepseek-sse.js` 的 `extractFragmentText` 在项目记忆中被标注为"隐式控制流与信号过滤耦合"。新增 Gemini 桥接如果引入新的 SSE 格式，会对此模块产生压力。这个耦合点在 Issue 分析中应作为固定检查项。
