## 深度反思

### 1. 审查质量评估
- **覆盖度**：审查聚焦于本次 PR 的核心变更——工具调用解析扩展、流式门控移除和调试日志。发现了致命的 `debugLog` 未定义问题、多段重复代码、条件死分支、索引越界隐患、敏感数据泄漏等关键点，对主要风险路径覆盖到位。  
  **潜在遗漏**：新增 12+ 个解析器的策略正确性与边界安全性（如 XML 注入、循环解析开销）未逐一深究，未评估这些解析器在恶意输入下的鲁棒性；`debug-logger` 的同步写入性能影响在白热路径中未提及。不过，首次全量审查难以穷尽所有细节，这些可作为后续增量审查的关注点。

- **准确度**：严重/主要问题标记基本合理。`debugLog` 调用未定义确属阻断性崩溃；条件死分支和索引-1 风险均为逻辑隐患。但 `logUpstream` 隐私问题在实际运维中可能是争议点（调试功能本身设计如此），标记为 major 略显严苛，可降为 minor 并建议添加脱敏选项。

- **完整性**：审查准确理解了 PR 意图（解决门控陷阱、提升兼容性），正面肯定了架构方向。但对变更的整体测试状态（如新增解析器是否有覆盖测试）未加评判，在决策 “request_changes” 时可补充要求补充测试用例，提升说服力。

### 2. 发现的模式
- **代码重复**：`checkForToolCallMarker` 在三桥接中相同实现，再次印证项目知识中“多桥接一致性”痛点，表明团队习惯在桥接中快速复制，缺乏及时抽象。
- **魔术值硬编码与维护脆弱性**：7 个 `text.includes(PREFIX)` 链式判断及信号文本 `"FINISHED"` 基于内容判别，是典型的“修改时易遗漏”模式。未来新增标记仍需手动更新排除条件，可靠性差。
- **防御性编程不足**：`indexOf` 返回 -1 直接传入解析器、环境变量控制函数定义删除后未全局扫描调用方，反映出缺少自动化检查（如 lint、单元测试）的工程习惯。
- **日志设计半成熟**：`debug-logger` 在闭包收集、零开销方面设计清晰，但硬编码路径、未区分日志级别的数据截断，说明在设计之初未充分考虑生产排障的隐私与部署需求。

### 3. 规范完善建议
基于本次审查，建议在项目记忆或审查规则中新增/强化以下条目：
- **函数删除影响扫描**：删除全局工具函数（如 `debugLog`）前，必须全文搜索所有调用点，或通过 ESLint `no-undef` 阻断构建。
- **多桥接共享逻辑提取**：任何在 ≥2 个桥接文件中出现的相同功能函数，必须提取到 `completion-core.js` 或新建共享模块，评审时将其标记为必要条件。
- **工具标记维护策略**：所有工具调用前缀标记应定义在 `MARKER_STRATEGIES` 配置数组中，相关排除/检测逻辑应通过数组驱动，禁止使用硬编码链式判断。新增标记时必须同步更新配置。
- **调试日志脱敏规则**：任何写文件的调试日志，不得完整记录用户输入或对话历史，必须对 prompt 内容实施截断或哈希化，并提供 `DEBUG_SANITIZE` 环境选项。
- **SSE 信号判别原则**：禁止仅凭文本内容（如 `"FINISHED"`）过滤数据帧，必须依据结构特征（如 `payload` 中特定字段缺失）或显式信号字段区分。
- **索引安全把关**：调用 `parseJsonFrom` 等起始位置函数时，须在传入前校验 `indexOf` 结果不为 -1；可将此规则纳入常用模式。

### 4. 经验教训
- **工具调用解析增强的审查重心**：面对大量新增解析器，不能只关注新增代码本身，更要审查其与旧排除条件的集成是否正确、是否会产生解析歧义或性能消耗。后续此类 PR 应强制要求提供多格式压力测试的日志或截图。
- **重构中删除代码的高危性**：任何看似“替换为更优实现”的删除操作，极容易遗留隐式引用。用 IDE 的“查找使用”或引入 TypeScript 的类型检查可有效规避。
- **环境变量与日志的边界**：调试开关应严格限定为诊断辅助，不得影响核心业务路径（如 `if(DEBUG) ... else ...` 逻辑）。本次 PR 若 `debugLog` 未被删除，而是作为条件输出，则不会崩溃，凸显了业务逻辑与调试输出解码耦的重要性。
- **多桥接架构的持续债务**：重复代码、不一致的流式后处理，表明项目需要尽早定义桥接基类或规范管道，否则每次兼容性增强都将成倍增加维护成本。建议在后续重构中优先解决此架构债。

综上，这次审查质量扎实，抓住了关键阻断缺陷和可维护性问题，但在全面测试要求和隐私分级上的评判可更精细。未来可借助自动化工具强化部分审查规则，减少人为疏漏。
