### 2. 发现的模式

**代码模式**  
- **常量逐步规范化**：从硬编码字符串到局部常量，再到风格统一的前缀命名，反映出项目正从“可行”向“可维护”过渡的迹象。审查建议推动以 `TOOL_CALL_` 前缀建立语义簇，可以显著降低误用风险。  
- **控制流澄清通过注释**：对 `extractFragmentText` 的隐式依赖用显式注释解决，符合“代码即文档”原则，适合资源有限的小型项目。未来更深层改进（如用卫士子句显式区分）可作为进阶建议。

**架构观察**  
- `completion-core.js` 正在成为工具调用解析逻辑的汇聚点，常量、标记、检测函数均集中于此。这有利于复用，但也可能让该文件膨胀为“工具调用工具包”。建议后续关注其职责边界，避免泛化。  
- `deepseek-sse.js` 的 SSE 解码逻辑与信号过滤耦合较紧，当前修改通过精确的 `typeof` 检查降低了过滤误判，说明解码与业务过滤的分离不够彻底。长期看可将 SSE 解码、帧类型识别、业务过滤分层处理。

### 3. 规范完善建议

基于本次审查，可新增/强化以下审查规则：

- **常量命名一致性规则**：当同一文件或同一语义域存在多个常量时，应使用统一前缀（如 `TOOL_CALL_*`）形成命名空间，审查时若发现孤立命名需提示。该规则可直接纳入“常见代码模式与审查要点”的**工具调用门控陷阱**子项。  
- **SSE 数据帧过滤条件编写规范**：要求过滤条件不仅正确，还应通过注释说明“信号帧的结构特征”（如 `payload.v` 在信号帧时是裸字符串），避免未来维护者因协议变更而错误放宽条件。  
- **增量审查中“遗留问题关闭确认”要求**：当前轮次已实施，但可强化为模板项：每轮必须逐条列出前轮问题及关闭证据，防止虚假修复。

### 4. 经验教训

- **隐式控制流的“注释修复”是双刃剑**：虽然注释改善了可读性，但控制流仍然脆弱。未来对类似场景（如 `extractFragmentText` 中的字符串优先处理）应建议采用显式的 `if/else if` 分支或提前返回结构，而非仅靠注释。审查时可将此作为可选的下一级优化建议。  
- **常量提取时要预判扩散风险**：即使当前模块内部使用，若常量具有通用语义（如工具调用 JSON 前缀），审查应提醒未来可能需要导出。非强制，但能让维护者提前规划 API 边界。  
- **条件精确化优于逻辑依赖**：本次用 `typeof payload.v === "string"` 替换 `!payload.v?.response?.fragments`，是典型的“从依赖数据结构细节到依赖类型特征”的进步。后续审查遇到类似依赖深层嵌套字段的过滤，应优先引导改用更稳定的类型/特征判断。
